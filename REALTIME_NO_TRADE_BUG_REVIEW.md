# 回测有成交但实盘/模拟盘零成交问题 Review

## 一、问题现象

当前系统在回测中可以看到有成交、有净值曲线、有回撤数据，但切到 MT5 模拟盘或真实盘后，长时间没有任何一笔真实成交。

这个问题容易让人误以为：

- 策略逻辑在实盘环境坏了
- 下单模块没有真正工作
- MT5 接口连接异常
- 回测结果与实盘结果完全不一致

实际排查后，这个问题不是单点 bug，而是“策略触发条件 + 实盘运行机制 + 风控过滤 + 样本认知偏差”共同造成的。

## 二、当前已确认的事实

### 1. 回测引擎本身不是完全失效

已验证回测模块可以真实产生成交，说明以下链路是通的：

- 策略会生成入场信号
- 风控模块会允许部分信号通过
- 回测成交与退出逻辑能正常执行
- 回撤与净值曲线并不是伪造数据

因此，“有回撤数据”本身就说明回测里确实发生过成交。

### 2. 实盘/模拟盘不是每根历史 K 线都会补单

实盘引擎只处理“新收盘 K 线”的最新信号，不会把你启动前已经出现过的历史信号拿来补做。

这意味着：

- 回测会扫完整段历史数据
- 实盘只看你启动后新出现的那一根已收盘 K 线

所以两者天然不是同一种运行方式。

### 3. XAU 的主要阻塞不是“没信号”，而是“信号被过滤”

对真实 XAU `M1` 数据做过诊断，结果是：

- 原始信号不是 0
- 但大量信号被 `outside_trading_window` 拦掉
- 还有一部分被 `consecutive_loss_limit_reached` 拦掉

这说明黄金策略不是完全不出信号，而是你的风控窗口把很多机会挡掉了。

### 4. BTC 不是高频策略

BTC 当前策略是 `M15` 趋势/突破型策略，本身就是低频：

- 它不是每几分钟都出手
- 需要趋势、突破、ADX、RSI、成交量等条件同时满足
- 启动后短时间无单，属于可能的正常现象

### 5. 样例数据曾经误导判断

此前样例文件存在两个典型问题：

- `sample_btc.csv` 实际是 `1 分钟` 间隔，但配置是 `M15`
- synthetic 样本结构过于平，不容易触发 breakout

因此，用这些样例去判断“策略是不是坏了”，结论会严重失真。

## 三、为什么会出现“回测有成交、实盘零成交”

## 结论

这不是单一 bug，而是以下几个原因叠加：

### 原因 1：实盘只处理新收盘 K 线，不追历史信号

回测是整段历史逐根重放。

实盘是：

- 拉取最新行情
- 只在出现新的已收盘 K 线时计算一次
- 只处理当下这一刻的新信号

如果你启动程序时，信号刚好已经过去，那么实盘不会补下。

### 原因 2：交易时段限制过严

黄金默认配置里有时段限制，例如：

- `14:00-02:00`

如果信号发生在这个窗口之外，日志就会看到：

- `New trade blocked: outside_trading_window`

这不是异常，而是配置在生效。

### 原因 3：连续亏损熔断 / 日内限制会阻止新开仓

如果当天已有连续亏损，或者达到日内亏损限制，那么新信号即便出现，也会被风控挡住。

### 原因 4：日内单方向限制可能进一步减少成交

如果开启 `one_direction_per_day=true`，当天先做多后，再出现做空信号时会直接被挡掉。

### 原因 5：BTC 策略本身低频

BTC `M15` 策略不是设计成高频交易，它要求多个条件同时成立，所以你短时间观察不到成交并不必然说明 bug。

### 原因 6：新闻黑窗也可能拦截

如果自动新闻文件可用，实盘会周期性刷新黑窗。

在黑窗时段内，系统不会新开仓。

### 原因 7：样本周期认知错误

如果你拿 `M1` 数据去验证 `M15` 策略，或者拿异常稀疏/异常平滑的样本去验证 breakout 策略，结论会偏离真实运行场景。

## 四、这是不是“成交模块 bug”

当前证据看，不应先把问题定性为“下单模块坏了”。

更准确的判断是：

- 下单模块可能暂时没有拿到可执行信号
- 或拿到信号后被风控挡掉
- 或者信号出现时程序根本还没启动

只有当满足以下条件时，才应进一步怀疑执行链路：

- 日志明确显示 `Opening buy position because ...` 或 `Opening sell position because ...`
- 但 MT5 端仍然没有任何订单

如果连这类日志都没有，问题优先级仍应放在“信号与过滤链路”，而不是“执行模块损坏”。

## 五、建议的解决流程

### 第一步：先判断有没有原始信号

运行：

```bash
mt5-quant diagnose-signals --config config.xauusd.m1.yaml --bars 1500 --output-dir reports/xau_diag
```

或：

```bash
mt5-quant diagnose-signals --config config.btcusd.m15.yaml --bars 2500 --output-dir reports/btc_diag
```

先看：

- `raw_entry_signal_count`
- `recent_entry_signals`

如果原始信号就是 0，那先别看执行模块。

### 第二步：看是否被风控拦截

重点看：

- `blocked_entries`

如果主要是：

- `outside_trading_window`
- `consecutive_loss_limit_reached`
- `news_blackout_window`
- `one_direction_per_day`

那就是配置限制问题，不是执行 bug。

### 第三步：把实盘观察窗口放到正确时段

例如黄金配置是上海时间 `14:00-02:00`，那你就不能在 `13:00` 左右期待系统开新仓。

### 第四步：确认实盘引擎只在新收盘 K 线处理信号

例如：

- `M1` 要等新 1 分钟 K 线收盘
- `M15` 要等新 15 分钟 K 线收盘

所以不要在 K 线还没收盘时判断“为什么没下单”。

### 第五步：如果确认有开仓日志但 MT5 没单，再查执行链路

这时再看：

- 账户权限
- 品种名是否正确
- 最小手数/步进
- 市场是否开盘
- 止损距离是否被券商拒绝
- MT5 `Experts` / `Journal` 日志

## 六、建议的工程化修复方向

为了让后续更容易定位，建议继续做以下增强：

### 1. 增加实盘信号诊断日志

在实盘每次处理新收盘 K 线时，除了记录最终是否开仓，还应记录：

- 当前 K 线时间
- 策略动作
- 策略原因
- 是否被风控拦截
- 风控拦截原因

### 2. 增加“只诊断不下单”的实盘观察模式

也就是：

- 接行情
- 生成信号
- 打日志
- 不真正下单

这非常适合排查“到底是没信号，还是信号被过滤”。

### 3. 增加 GUI 一键诊断入口

让用户不用命令行就能直接导出：

- 最近 N 根 K 线信号诊断报告
- 原始信号与拦截统计

### 4. 增加“最近一次未成交原因”状态展示

例如 GUI 或日志里直接显示：

- 最近一次原始信号时间
- 最近一次未成交原因

## 七、最终结论

当前这个“回测有成交、实盘零成交”的问题，优先结论不是“程序根本不会下单”，而是：

- 回测与实盘运行模型不同
- 黄金大量信号被时间窗和风控过滤
- BTC 策略本身低频
- 程序只处理启动后新收盘 K 线
- 样本周期和验证方式容易误导判断

因此，正确解决路径不是盲目改策略执行器，而是：

1. 先用 `diagnose-signals` 确认信号真实存在
2. 再确认是否被配置拦截
3. 再确认观察时段和 K 线收盘时机是否正确
4. 只有在“明确尝试开仓但 MT5 未成交”时，才进入执行链路排查
